## 一、存储架构概览

附件采用 **Kubernetes 持久化存储卷（PVC）** 方案，独立于应用容器存在。即使应用更新、重启或迁移，附件文件均不会丢失。

```
用户上传附件
    ↓
应用容器（/app/uploadPath/）
    ↓（挂载）
持久化存储卷 PVC（30 GB）
    ↓（存储在）
服务器磁盘（/dev/vda3，148 GB）
```

### 4.3 命名规则说明

| 要素 | 规则 |
|------|------|
| 磁盘文件名 | 采用 UUID（32位随机字符串 + 原始扩展名），避免重名覆盖 |
| 下载时显示名 | 保留用户上传时的原始文件名（存储在数据库中） |
| 业务名称中的特殊字符 | `/` `\` `:` `*` `?` `"` `<` `>` `\|` 自动替换为 `_` |
| 日期格式 | `yyyyMMdd`，取文件上传日期 |

## 六、数据安全与备份

| 保障措施 | 说明 |
|----------|------|
| 持久化存储 | 附件存储在独立 PVC 中，应用重启/更新不影响文件 |
| 操作审计日志 | 每次上传、下载、删除均记录操作人、时间、文件名（`pm_attachment_log` 表） |
| 软删除 | 附件删除仅标记状态（`del_flag`），不立即清除物理文件，可追溯恢复 |
| 数据库记录 | 文件元数据（名称、大小、类型、路径）存储在 `pm_attachment` 表 |

> **建议：** 定期将服务器 `/var/lib/rancher/k3s/storage/pvc-*_newpm_upload-pvc/` 目录备份至对象存储（如 OSS/MinIO），以防硬件故障导致数据丢失。
