# 项目信息管理系统 — 需求文档

> 版本：V1.4 | 最后更新：2026-03-31
> 基于代码实现 + `docs/gen-specs/*.yml` + 源码 Controller/Domain 整理

---

## 一、项目概述

### 1.1 背景

企业需要对软件项目全生命周期进行精细化管理，涵盖：立项申请与审批、合同签订与款项跟踪、人员日报工时、收入确认、任务分解与排期等环节。系统基于 **RuoYi-Vue v3.9.1** 框架定制开发。

### 1.2 系统角色

| 角色 | 代码 | 职责 |
|------|------|------|
| 超级管理员 | `admin` | 系统全权管理 |
| 项目经理 | `pm` | 立项申请、项目维护、日报填写 |
| 市场经理 | `scjl` | 立项申请、客户管理 |
| 销售负责人 | `xsfzr` | 合同相关操作 |
| 团队经理/部门经理 | — | 项目审批、项目经理变更、收入确认 |
| 普通员工 | — | 日报填写 |

### 1.3 部门概念

部门树级别（从大到小）：公司 → 部门 → 子部门 → 项目团队 → 项目小组

页面中"项目部门"选择器仅显示**三级及以下**机构，一二级不出现。三级部门作为根节点，四五级部门依次挂载。选中任意节点后端自动查询本级及所有子级数据。

## 二、功能模块需求

### 2.1 项目管理

#### 2.1.1 立项申请

**业务说明**：项目经理或市场经理提交新项目立项申请，项目初始审核状态为"待审核"。

**项目编号规则**：`{行业代码}-{一级区域代码}-{二级区域代码}-{简称}-{立项年度}`

**表单字段**：

*一、项目基本信息*

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 行业 | 下拉选 | ✓ | 字典：`industry` |
| 一级区域 | 下拉选 | ✓ | 字典：`sys_yjqy` |
| 二级区域 | 下拉选 | ✓ | 关联一级区域动态显示，数据源：`pm_secondary_region` |
| 简称 | 文本 | — | 项目简称 |
| 立项年度 | 下拉选 | ✓ | 字典：`sys_ndgl` |
| 项目编号 | 只读 | — | 系统自动生成 |
| 项目名称 | 文本 | ✓ | — |
| 项目分类 | 下拉选 | ✓ | 字典：`sys_xmfl` |
| 项目部门 | 树形选择 | ✓ | 三级及以下部门 |
| 项目状态 | 下拉选 | ✓ | 字典：`sys_xmzt` |
| 验收状态 | 下拉选 | ✓ | 字典：`sys_yszt` |
| 预估工作量 | 数字 | ✓ | 单位：人天 |
| 实际工作量 | 只读 | — | 由日报自动汇总，单位：人天 |
| 项目预算 | 数字 | ✓ | 单位：元 |
| 项目地址 | 文本 | ✓ | — |
| 项目计划 | 文本域 | ✓ | — |
| 项目描述 | 文本域 | ✓ | 服务类项目填写巡检信息 |

*二、人员配置*

| 字段 | 类型 | 必填 | 数据源 |
|------|------|------|--------|
| 项目经理 | 下拉选 | ✓ | 岗位编码 `pm` 对应用户昵称 |
| 市场经理 | 下拉选 | ✓ | 岗位编码 `scjl` 对应用户昵称 |
| 参与人员 | 多选 AutoComplete | ✓ | 所有用户昵称，已选显示为 Tag |
| 销售负责人 | 下拉选 | ✓ | 岗位编码 `xsfzr` 对应用户昵称 |
| 销售联系方式 | 只读 | ✓ | 选择销售负责人后自动带出 |

*三、客户信息*

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 客户简称 | 下拉选/搜索 | ✓ | 数据源：`pm_customer` |
| 客户联系人 | 下拉选 | ✓ | 关联客户动态显示 |
| 客户联系方式 | 只读 | — | 选择联系人后自动带出 |
| 商户联系人 | 文本 | — | — |
| 商户联系方式 | 文本 | — | — |

*四、时间规划*

启动日期、结束日期、投产日期、验收日期（均为日期选择器）

*五、成本预算*

项目费用、费用预算、成本预算、人力费用、采购成本（均为数字输入，单位：元）

*六、备注*

**保存行为**：
- 系统自动同步项目经理、市场经理、销售负责人、参与人员到 `pm_project_member`
- 自动初始化：收入确认年度 = `dd`（待定），收入确认状态 = `0`（未确认）

#### 2.1.2 立项审核

**描述**：团队经理岗位人员审核待审核的项目立项申请。

**查询条件**：项目名称、项目编号、项目部门、行业、项目分类、一级区域、二级区域、项目经理、市场经理、项目阶段、审核状态

**列表字段**：项目编号、项目名称、项目部门、项目分类、一级区域、二级区域、项目阶段、审核状态、项目经理、市场经理、立项年度、创建时间、操作（审核）

**审核详情**：抽屉展示，包含项目全部信息 + 审核状态 + 审核意见

**审核操作**：
- 通过：更新 `pm_project.approval_status = 1`，写入审核记录到 `pm_project_approval`
- 拒绝：必须填写审核意见，更新 `pm_project.approval_status = 2`
- 退回待审核：审核状态回退为 0，写审核记录

**状态说明**：0=待审核，1=审核通过，2=审核拒绝，3=退回待审核

---

#### 2.1.3 项目管理（列表）

**查询条件**：项目名称（AutoComplete）、项目部门、收入确认年度、项目分类、一级区域、二级区域、项目经理、市场经理、立项年度、项目阶段、审核状态、验收状态、创建人、最后更新日期、最后更新人

**列表字段**：序号、项目名称、项目部门、项目经理、项目分类、项目预算、预估工作量（人天）、实际人天、合同金额、收入确认年度、合同状态、收入确认状态、确认金额、参与人员（姓名+实际人天）、启动日期、结束日期、验收日期、审核状态、项目状态、验收状态、更新人、更新时间、操作

**列表顶部汇总行**：在表头下方、数据行上方展示各数值字段合计，操作列无按钮

**操作**：查看详情、编辑、添加合同/查看合同、收入确认、附件、删除、打印立项申请单

**人天展示公式**：`ROUND(actual_workload / 8, 3) + COALESCE(adjust_workload, 0)`（禁止直接展示原始小时数）

---

#### 2.1.4 项目详情

展示分区：

1. **项目基本信息**（左侧）：项目名称、项目ID、项目分类、行业、区域、项目部门、项目状态、项目阶段、确认日期、审核状态、审核意见、立项年度、收入确认年度、项目编号、项目地址、项目计划、项目描述

2. **人员/客户/时间/成本**（右侧）：项目经理、市场经理、参与人员（姓名+实际人天）、销售负责人、客户信息、各时间节点、成本预算各项

3. **合同信息**：合同名称（超链接）、合同编号、合同状态、合同类型、签订日期、合同周期、合同金额、不含税金额、税率、免维期、创建/更新审计字段

4. **款项信息**：付款方式、里程碑要素、违约扣款、付款状态、季度、日期、备注、审计字段

5. **项目关联任务列表**：任务编号、投产批次、产品、任务名称、预估工作量、功能测试版本日期、计划投产日期、任务负责人、排期状态

项目名称、合同名称、任务名称均支持右键新标签打开及超链接跳转详情。

#### 2.1.5 项目编辑

字段与立项申请相同，额外回显：审核状态、审核意见。项目编号不可修改。

**允许扩展名**：`doc, docx, xls, xlsx, pdf, csv, png, jpg, gif, txt, 7z, zip, gz`

**查询条件**：项目名称（模糊）、项目经理（下拉，模糊）

**列表字段**：项目名称、当前项目经理、创建日期、创建人、最后更新日期、最后更新人

**操作**：批量修改（选择新项目经理，下拉选，模糊查询）、重置

#### 2.1.8 项目人员管理

**查询条件**：项目部门、项目名称

**列表字段**：项目名称、参与人员

#### 2.1.9 公司收入确认

**查询条件**：同项目管理查询条件 + 收入确认状态

**列表字段**：同项目管理列表 + 确认金额、税后金额、确认日期

**列表汇总**：数值列合计行

**操作**：查看、编辑（收入确认要素）、导出

**收入确认要素**：
- 左侧：项目详情（基本信息，不含备注和5个备用字段）
- 右侧：收入确认状态（字典：`sys_qrzt`）、收入确认年度（字典：`sys_ndgl`）、确认日期、确认金额（含税）、税率（6%软件开发/13%硬件）、税后金额（= 确认金额 ÷ (1 + 税率)）、备注

**收入确认状态**（`sys_qrzt`）：未确认、待确认、已确认、无法确认

#### 2.1.10 团队收入确认

**数据来源**：以 `pm_project` 为主表，LEFT JOIN `pm_team_revenue_confirmation`

**特殊过滤**：`confirmDeptId = -1` 表示查询无团队确认的项目

**操作**：批量确认、查看公司收入确认相关接口（需自动授权）

### 2.2 合同信息管理

#### 2.2.1 合同管理

**查询条件**：合同名称、合同所属部门（树形组件）、合同类型、关联客户

**列表字段**：序号、合同名称、关联项目、客户简称、合同所属部门、合同类型、合同签订日期、合同周期、合同金额、不含税金额、免维期、备注、创建日期、创建人、最后更新日期、最后更新人

**操作**：删除、附件管理

**新增/编辑字段**：
- 部门（必填）
- 合同名称（必填）
- 关联客户（必填，下拉选）
- 关联项目（下拉选，AutoComplete）
  - 编辑合同时：关联项目后面添加项目所属部门
- 合同类型（必填，字典：`sys_htlx`）
- 合同签订日期（必填，YYYY-MM-DD）
- 合同周期（月）
- 合同编号
- 合同金额（必填，含税）
- 税率（%）：6%（软件）/ 13%（硬件）
- 不含税金额（= 合同金额 ÷ (1 + 税率)）
- 税金（= 不含税金额 × 税率）
- 免维期（月，必填）
- 备注

**详情**：合同基本信息与合同时间/周期左右两列布局，含审计字段。详情下方展示关联任务列表。

**合同名称**支持右键新标签打开。

**操作**：查看、编辑、删除、附件管理、导出

**新增/编辑字段**：付款方式名称、里程碑要素（名称+金额，多组）、是否违约扣款、扣款金额、付款状态、预计回款季度、实际回款季度、开票日期、实际回款日期、款项确认年份、备注

**合同所属团队过滤规则**：树只显示三级及以下部门，三级为根节点，支持多选，选中任意节点查询本级及所有子级数据。

### 2.4 日报管理

#### 2.4.1 日报填写

**描述**：员工每日填写工作日报，记录当日项目工时和工作内容。

**页面布局**：
- **左侧（7/24）**：MonthCalendar 组件，日期格子展示 `total_work_hours`（绿色 ≥8h，橙色 <8h，灰色无数据），点击日期加载当日详情
- **右侧（17/24）**：当前日报编辑区

**编辑区 — 假期管理（上方）**：

点击"添加假期"，假期类型来自字典（`sys_gzlb` 子集），支持请假、倒休、年假，也支持婚嫁、丧家、产假、哺乳假等（规划中）。非工作类条目使用 `leave_hours` 字段。

**编辑区 — 项目工时（下方）**：

默认显示当前用户参与且状态为"开启"、阶段未到"项目结项"的项目。

对于**普通项目**（无子任务）：

| 展示字段 | 说明 |
|----------|------|
| 项目经理、收入确认年度 | 项目信息回显（收入确认年度非当年显示红色） |
| 工时滑块 | 可手动拖动 |
| 工作任务类别（必选，多选） | 字典：`sys_gzlb` |
| 工作内容文本域 | — |

对于**有任务的项目**（`hasSubProject=true`）：
- 展示项目信息行
- 展示任务行（每个任务一行）：批次号、任务负责人、任务名称、任务当前阶段、预计人天、实际人天、工作任务类别（必选，多选）、工时、工作内容
- 项目工时由任务工时汇总而来，项目行不可手动拖动工时

**周限制**：仅当前自然周（周一至周日）可编辑，过去/未来周只读

**提示**：日历下方显示"项目状态非开启或阶段属于'已结项'则不可填日报；仅某任务已结项则该任务不可显示"

**保存**：`POST /project/dailyReport`，保存后重新计算子任务 `actual_workload` 并汇总到父项目

**批量请假**：`POST /project/dailyReport/batchLeave`，支持一次性为连续日期批量填写假期条目（请假/倒休/年假等），避免逐日手动添加。

#### 2.4.2 工作日报动态

**查询条件**：日报日期（开始/结束范围）、姓名（下拉，模糊）、项目部门（下拉选）

**页面展示**：按日历卡片显示（非列表），卡片按月显示，格子内展示该人当日日报信息

**日报详情抽屉**（宽度较大）：

- 项目信息行：收入确认年度、项目经理、项目名称、项目当前阶段、预计人天、已花人天、工作任务类别（收入确认年度非当年标红）
- 任务信息行（如有）：任务编号、任务负责人、任务名称、任务当前阶段、预计人天、实际人天、工作任务类别
- 项目名称、任务名称均支持超链接跳转详情

**颜色规则**：当日工时 ≥8h 绿色，<8h 橙色

**项目经理可见性**：项目经理可查看不属于本团队但参与了该项目的人员日报

**已填写/未填写人员统计**：`GET /project/dailyReport/activityUsers`，按日期范围统计部门内已提交和未提交日报的人员清单

---

#### 2.4.3 项目人天统计

**查询条件**：所属团队（可具体到组）、收入确认年份、项目名称

**页面展示**（表格设计，非纯列表）：项目名称、所属团队、团队负责人、预计人天、项目总实际人天、成员姓名及其各自实际人天

---

#### 2.4.4 团队日报

**查询条件**：项目部门（三级及以下树）、项目名称（AutoComplete）、日期（年月）、收入确认年度

**页面展示**（日历卡片，非纯列表）：部门、项目名称、姓名、日历卡、日报内容、工时

**团队信息**：是否带来收入的项目（有关联合同则是，可用颜色区分）；实际人天 = 项目表中的人天 + 调整人天；与预算人天对比（实际尽量不超预算的一半）

**数据权限**：RuoYi 数据权限过滤

**菜单**：日报管理一级菜单下，第5个二级菜单，图标 date

**项目筛选下拉**：`GET /project/dailyReport/teamProjectOptions`，根据部门、收入确认年度获取可选项目列表（供团队日报页面 AutoComplete 使用）

---

#### 2.4.5 日报填报统计（周统计）

**描述**：按周统计各部门人员的日报填报情况，区分已提交与未提交。

**查询条件**：项目部门（三级及以下树）、自然周选择（自动计算 startDate/endDate）

**部门树**：`GET /project/dailyReport/weeklyStatsDeptTree`，返回当前用户数据权限范围内的部门树

**页面展示**：

| 列 | 说明 |
|----|------|
| 姓名 | 员工姓名 |
| 部门 | 所属部门 |
| 周一~周日 | 每天的工时小时数（绿色 ≥8h，橙色 <8h，灰色未填） |
| 合计 | 本周总工时 |

**明细查看**：`GET /project/dailyReport/weeklyStatsDetail`，点击"已提交"/"未提交"人数，展开查看具体人员清单。参数 `type=submitted` 或 `type=unsubmitted`。

#### 2.4.6 日报白名单管理

**描述**：将特定人员加入免填日报白名单，白名单中的人员不纳入"未提交"统计。典型场景：长期出差人员、高管等。

**查询条件**：关键词（姓名/部门模糊搜索）

**列表字段**：姓名、部门、加入原因、创建时间、操作

**操作**：
- **添加**：选择用户（`userId`，必填）、填写加入原因（`reason`，必填）
- **移除**：从白名单中删除（软删除）
- **自检**：`GET /project/whitelist/checkSelf`，当前用户检查自己是否在白名单中（无需权限）

**权限**：`project:whitelist:list`、`project:whitelist:add`、`project:whitelist:remove`

**菜单**：日报管理一级菜单下

#### 2.4.7 工作日历管理

**描述**：维护年度工作日历，标记节假日和调休工作日。日报模块据此判断某天是否需要填写日报。

**查询条件**：日期范围（开始/结束）、年份

**列表字段**：日期、类型（holiday/workday）、名称（如"春节"）、年份

**操作**：
- **单条新增/编辑/删除**：标准 CRUD
- **按年批量保存**：`POST /project/workCalendar/batchSave`，传入某年所有特殊日期（先删除该年全部记录，再批量插入）
- **按年查询**：`GET /project/workCalendar/year/{year}`，返回该年所有日历条目（不分页，供 MonthCalendar 组件渲染）

**日历类型说明**：
- `holiday`：法定节假日，无需填写日报
- `workday`：调休工作日（如周末补班），需要填写日报

**权限**：`project:workCalendar:list`、`project:workCalendar:add`、`project:workCalendar:edit`、`project:workCalendar:remove`

**跨模块权限**：`/year/{year}` 端点同时允许 `project:dailyReport:list` 和 `project:dailyReport:activity` 权限访问（日报和动态页面需要渲染日历）

## 三、非功能需求

### 3.1 权限与安全

- 所有接口必须通过 JWT 鉴权（公开接口使用 `@Anonymous`，不得滥用）
- 所有 Controller 接口必须有 `@PreAuthorize` 权限注解
- 涉及部门数据隔离的接口必须使用 `@DataScope` 注解
- PM 角色用户禁止直接调用 `system/` 前缀接口，必须通过代理接口：
  - 用户列表：`GET /project/project/users?postCode=xxx`
  - 部门树：`GET /project/project/deptTree`（数据隔离）/ `deptTreeAll`
  - 客户列表：`GET /project/project/customers`

### 3.2 审计日志

- 所有新增/修改/删除接口必须添加 `@Log(title, businessType)` 注解
- 项目状态/阶段变更必须写入 `pm_project_stage_change`
- 附件操作（上传/下载/删除）必须写入 `pm_attachment_log`
- 人天补正操作写入 `pm_workload_correct_log`
- `syncProjectMembers()` 操作不得更新 `pm_project.update_by/update_time`

### 3.3 性能

- 列表分页，默认每页 10 条
- 项目列表支持服务端排序（`orderByColumn` + `isAsc`）
- 字典数据 Redis 缓存

### 3.4 文件上传

- 最大 30 MB
- 允许扩展名：`doc, docx, xls, xlsx, pdf, csv, png, jpg, gif, txt, 7z, zip, gz`
- 存储路径：`{业务类型}/{ID}_{名称}/{yyyyMMdd}/{UUID}.{ext}`

### 3.5 数据删除策略

| 删除方式 | 适用表 |
|----------|--------|
| **硬删除** | `pm_project`、`pm_task`、`pm_production_batch`、`pm_daily_report`、`pm_daily_report_detail` |
| **软删除**（`del_flag='1'`）| 其他所有 PM 业务表 |
