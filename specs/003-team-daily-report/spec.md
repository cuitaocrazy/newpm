# Feature Specification: 团队日报

**Feature Branch**: `003-team-daily-report`
**Created**: 2026-03-19
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 按部门+月份查看团队日报概览 (Priority: P1)

项目经理或管理人员选择部门和年月，查看该部门下所有项目参与人员的月度日报全貌——谁在做哪个项目、每天填了多少工时、项目是否带来收入。

**Why this priority**: 这是功能的核心价值：从团队视角而非个人视角审视日报，是管理层最频繁的使用场景。

**Independent Test**: 选择部门+月份后，页面展示日历卡片区域，每个人员/项目对应一行，各日期格显示工时徽章；带合同项目行用绿色高亮。

**Acceptance Scenarios**:

1. **Given** 当前用户有部门数据权限，**When** 选择三级部门 + 2026年3月，**Then** 展示该部门所有项目成员的月历卡，无权限部门数据不出现
2. **Given** 选中四级部门，**When** 提交查询，**Then** 后端自动包含该四级部门本身（ancestors 层级逻辑）
3. **Given** 某项目有关联合同，**When** 渲染该项目行，**Then** 行背景/标识为绿色；无合同项目为默认色

---

### User Story 2 - 按项目名称筛选聚焦 (Priority: P2)

用户在项目名称输入框输入关键词，autocomplete 下拉候选，选中后列表只显示该项目相关人员的日报。

**Why this priority**: 部门下项目可能很多，需要快速聚焦到单个项目。

**Independent Test**: 输入项目名关键词后，下拉出现候选项；选中后日历卡区域仅保留该项目的人员行。

**Acceptance Scenarios**:

1. **Given** 已选部门，**When** 输入项目名称片段，**Then** autocomplete 返回该部门下匹配项目（≤20条）
2. **Given** 选中项目，**When** 渲染列表，**Then** 仅显示该项目的参与成员行

---

### User Story 3 - 实际人天 vs 预算人天对比预警 (Priority: P3)

在人员行尾显示「实际人天 / 预算人天」统计，当实际人天超过预算人天一半时，给出醒目的视觉提示。

**Why this priority**: 帮助管理层发现人力投入异常，是辅助决策的关键指标，但不影响基础查看功能。

**Independent Test**: 找一个实际人天已超预算50%的项目成员行，确认数值颜色变红或出现警告图标。

**Acceptance Scenarios**:

1. **Given** 某成员在项目的实际人天 ≤ 预算人天×0.5，**When** 渲染统计列，**Then** 数值正常显示（绿色或中性色）
2. **Given** 实际人天 > 预算人天×0.5，**When** 渲染，**Then** 统计数值标红或加警告标识

---

### Edge Cases

- 当月无任何日报数据时，日历卡区域显示空状态提示，而不是空白页面
- 某成员在同一天填了多个项目的日报，日历格需合并展示总工时
- 项目无预算人天（为0或NULL）时，跳过人天对比逻辑，不显示对比列
- 部门树只显示三级及以下，一二级部门完全不作为节点出现
- 选中三级部门节点，查询自动包含其下所有四级子部门

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供项目部门下拉树，**只显示三级及以下部门**，三级作为根节点，四级挂在下面；一二级部门完全不展示
- **FR-002**: 部门选择后，后端查询 MUST 自动包含选中部门及其所有子级（`find_in_set` ancestors 模式）
- **FR-003**: 项目名称输入框 MUST 使用 autocomplete，按当前选中部门范围在后端模糊搜索项目名称
- **FR-004**: 日期筛选 MUST 为「年月」粒度（月份选择器），默认当前月
- **FR-005**: 主展示区 MUST 采用**日历卡片式布局**（非纯列表）：行=人员×项目，列=月内各日期，格内显示当日工时
- **FR-006**: 是否带来收入项目的判断标准：**该项目是否存在关联合同**（`pm_project_contract_rel`），用颜色区分（有合同=绿色标识，无合同=默认）
- **FR-007**: 每行末尾 MUST 显示「实际人天」和「预算人天」对比；实际人天 = `ROUND(actual_workload/8, 3) + COALESCE(adjust_workload, 0)`
- **FR-008**: 当实际人天 > 预算人天 × 0.5 时，MUST 给出视觉预警（红色/警告图标）
- **FR-009**: 所有查询 MUST 应用若依数据权限过滤（`@DataScope` + `${params.dataScope}`）
- **FR-010**: 菜单位于「日报管理」一级菜单下，作为第5个二级菜单，图标为 `date`
- **FR-011**: 前端必须使用 `<project-dept-select />` 或等效代理接口，禁止直接调用 `system/dept`

### Key Entities

- **DailyReport** (`pm_daily_report` + `pm_daily_report_detail`): 日报主表和明细，按 `report_date` + `user_id` 关联
- **Project** (`pm_project`): 提供 `estimated_workload`（预算人天×8）、`actual_workload`、`adjust_workload`
- **ProjectContractRel** (`pm_project_contract_rel`): 判断项目是否有关联合同（EXISTS 子查询）
- **ProjectMember** (`pm_project_member`): 确定某人属于哪个项目
- **SysDept** (`sys_dept`): 部门层级，`ancestors` 字段支持层级查询；过滤三级及以下（ancestors LIKE 层级计数）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 选择部门+年月后，页面在 3 秒内渲染完成日历卡片区域（正常数据量）
- **SC-002**: 日历卡准确反映当月每日工时，与日报填写页数据一致
- **SC-003**: 有合同项目与无合同项目视觉区分清晰，用户无需查文档即可理解含义
- **SC-004**: 人天预警（超50%）标识在列表中显眼，管理员可快速识别超量成员
- **SC-005**: 数据权限生效——非本部门数据在列表中不出现
- **SC-006**: 三级以上部门（一、二级）在部门树中完全不可见
