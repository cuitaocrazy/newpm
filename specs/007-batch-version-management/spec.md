# Feature Specification: 批次版本管理（出入库版本）

**Feature Branch**: `007-batch-version-management`
**Created**: 2026-06-10
**Status**: Draft
**Input**: User description: "批次版本管理：出入库版本记录的增删改查、Excel 导出、出入库版本号按 6 种版本类型自动生成、多任务级联、关联 pm_task 与 pm_production_batch。试水迁移自老 yadapm 系统 /storage 页面。数据迁移仅限本菜单涉及的数据。"

> 背景：本功能迁移自老 yadapm 系统（Scala + Oracle）的"出入库版本管理 → 批次版本管理"页面（`/storage/*`）。这是 yadapm 整体迁移的第一块"试水"，跑通后再迁其余 4 个菜单。完整功能盘点见 `docs/plans/2026-06-10-yadapm-migration-requirements.md`。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 新增批次版本并自动生成出入库版本号 (Priority: P1)

版本管理人员在一个投产批次发布软件版本时，需要登记一条"批次版本"记录：选择投产年份、投产批次、产品、子系统、版本类型、组包方式等信息，关联一个或多个软件中心任务，系统据此**自动生成唯一的出入库版本号**并保存。

**Why this priority**: 这是整个模块的核心价值与最高复杂度所在。出入库版本号的自动生成（6 种版本类型各有不同规则）是老系统最难、最易出错的逻辑，必须最先打通。没有它，本模块没有存在意义。

**Independent Test**: 单独实现"新增"即可独立验证——录入一条批次版本，确认系统按版本类型规则生成正确的出入库版本号并落库，即交付了可用价值。

**Acceptance Scenarios**:

1. **Given** 用户选择了投产年份、批次、产品、子系统和版本类型"SP升级包"，**When** 提交新增，**Then** 系统生成形如 `{基准版本号}_SP{序号}` 的出入库版本号（序号为同子系统同类型当前最大序号+1，2 位补零），并保存记录。
2. **Given** 用户选择版本类型"临时版本包"，**When** 提交新增，**Then** 系统生成形如 `T_{当前年份}_{序号3位}_{产品}` 的版本号（序号按子产品+年份维度递增）。
3. **Given** 用户选择版本类型"B包升级包"且指定了升级包初级版本号，**When** 提交新增，**Then** 系统生成形如 `{基准版本号}_B{主}.{次}` 的版本号（次版本号在已有基础上递增，否则从 01 起）。
4. **Given** 用户选择了某投产批次，**When** 批次确定后，**Then** 版本投产日期自动带出且不可手动修改。
5. **Given** 用户为一条版本关联了多个软件中心任务，**When** 录入任务号，**Then** 系统回显对应的任务名称、项目名称、需求名称，且支持动态增删任务行。

---

### User Story 2 - 查询、筛选与查看批次版本列表 (Priority: P2)

版本管理人员需要按多种条件检索已登记的批次版本，并查看单条记录的完整详情。

**Why this priority**: 登记之后必须能查得到、看得清，是日常使用的高频路径。依赖 P1 已有数据，故次于 P1。

**Independent Test**: 在已有版本数据的前提下，按投产年份/批次/产品/任务号/版本类型/提交人员等条件查询，列表正确过滤、分页、可进入详情，即独立可用。

**Acceptance Scenarios**:

1. **Given** 列表存在多条版本记录，**When** 用户按投产年份+产品筛选，**Then** 仅返回匹配记录并分页展示。
2. **Given** 用户在结果列表中点击某记录，**When** 进入详情，**Then** 展示该版本的全部字段（含出入库版本号、关联任务、组包方式、版本状态等）。
3. **Given** 用户离开列表进入详情再返回，**When** 返回列表，**Then** 之前的查询条件被保留（不丢失筛选状态）。

---

### User Story 3 - 编辑与删除批次版本 (Priority: P3)

版本管理人员需要修订或删除已登记的批次版本记录。

**Why this priority**: 修订/删除是数据维护的补充能力，频率低于新增与查询，但属于完整 CRUD 的必要部分。

**Independent Test**: 选中一条已存在版本，修改字段后保存、或删除，确认变更落库并被审计记录，即独立可用。

**Acceptance Scenarios**:

1. **Given** 一条已存在的批次版本，**When** 用户修改非关键字段（如备注、版本说明）并保存，**Then** 记录更新，出入库版本号保持不变。
2. **Given** 一条已存在的批次版本，**When** 用户修改了影响版本号的关键字段（子系统/版本类型）并保存，**Then** 系统按规则重新生成出入库版本号。
3. **Given** 用户对一条版本执行删除，**When** 确认删除，**Then** 记录被移除且该操作被写入审计日志。

---

### User Story 4 - 导出批次版本为 Excel (Priority: P3)

版本管理人员需要把当前查询结果导出为 Excel 文件，用于线下汇报或归档。

**Why this priority**: 导出是常见但非核心的辅助能力，依赖查询结果存在。

**Independent Test**: 在有查询结果时点击导出，得到包含列表字段的 Excel 文件，即独立可用。

**Acceptance Scenarios**:

1. **Given** 用户已执行查询得到结果集，**When** 点击导出，**Then** 系统生成包含当前筛选结果全部字段的 Excel 文件供下载。

---

### User Story 5 - 迁移老系统批次版本存量数据 (Priority: P2) — ⏸️ 延后（阻塞：等待旧库最新数据）

> **状态：本轮不实现。** 目前尚未拿到旧数据库的最新数据，数据迁移与功能开发解耦：**先实现功能（P1–P4），数据迁移待旧库数据到位后单独执行**。本故事的需求与验收保留在此，作为后续迁移的依据，不计入本次实现范围。

将老 yadapm 系统中"批次版本管理"对应的存量数据导入新系统，使迁移后历史版本可在新系统中查询。

**Why this priority**: 试水迁移的目标之一是验证数据可迁移、可查询；没有存量数据，新系统对业务方而言是"断档"的。范围严格限定：**仅迁移批次版本管理这一个菜单涉及的数据，其余 yadapm 功能数据一律不迁**。功能本身可在空表上独立开发与验证，不依赖存量数据，故迁移可安全延后。

**Independent Test**: 执行迁移后，在新系统的批次版本列表中能查到老系统的历史批次版本记录，关键字段（出入库版本号、投产批次、产品、任务、提交人员）正确对应。

**Acceptance Scenarios**:

1. **Given** 老系统存在批次版本记录，**When** 执行数据迁移，**Then** 这些记录出现在新系统批次版本列表中，且出入库版本号等关键字段值与老系统一致（迁移保留原值，不重新生成）。
2. **Given** 老系统中同一张表里还混存着非批次版本数据，**When** 执行迁移，**Then** 仅迁移属于"批次"的记录，非批次记录不被导入本功能。

---

### Edge Cases

- 当同一子系统+版本类型下已存在版本号时，序号必须严格递增、无重复、无跳号冲突。
- 当用户未选择投产批次却尝试提交时，应阻止保存并提示必填。
- 当关联的软件中心任务号在任务库中不存在时，如何处理（拒绝/提示/允许留空）。
- 当老系统迁移数据中存在新系统字典/任务/批次里找不到的对应项时，如何落地（保留原文/标记缺失）。
- 当版本类型为"B包升级包/SP包升级包"但未提供升级包初级版本号时，版本号如何生成或是否阻止。
- 并发新增同子系统同类型版本时，序号生成是否会产生重复（需保证生成的原子性/唯一性）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 允许有权限的用户新增一条批次版本记录，包含投产年份、投产批次、子产品、子系统、版本类型、组包方式、是否涉及 TWS 改造、数据库是否修改、接口是否修改、版本说明、备注等字段；版本状态本轮为可选字段（其枚举值依赖生产数据，待数据到位后填充）。
- **FR-002**: 系统 MUST 根据版本类型（SP升级包/PTF补丁包/B测试包/临时版本包/B包升级包/SP包升级包）按各自规则**自动生成出入库版本号**，用户不可手动编辑该号。
- **FR-003**: 系统 MUST 在投产年份选定后联动可选批次，在产品选定后联动可选子系统，在子系统+版本类型选定后联动可选/计算升级包初级版本号。
- **FR-004**: 系统 MUST 支持一条版本关联多个软件中心任务，并在录入任务号时回显任务名称、项目名称、需求名称，支持动态增删任务行。
- **FR-005**: 系统 MUST 在选定投产批次后自动带出版本投产日期，且该日期对用户只读。
- **FR-006**: 系统 MUST 支持按投产年份、投产批次、产品、软件中心任务号、版本类型、提交人员、基准版本号、组包方式、版本状态、投产日期等条件分页查询批次版本列表。
- **FR-007**: 系统 MUST 提供单条批次版本的详情查看，展示其全部字段。
- **FR-008**: 系统 MUST 允许有权限的用户编辑已存在的批次版本；当影响版本号的关键字段变更时 MUST 按规则重新生成出入库版本号，否则保持原号不变。
- **FR-009**: 系统 MUST 允许有权限的用户删除批次版本记录。
- **FR-010**: 系统 MUST 支持将当前查询结果导出为 Excel 文件。
- **FR-011**: 系统 MUST 对新增、编辑、删除操作记录操作审计日志（操作人、时间、操作类型）。
- **FR-012**: 系统 MUST 对查询、编辑、删除等操作进行权限校验，无权限用户不可执行对应操作。
- **FR-013**: 系统 MUST 保证出入库版本号在同子系统+版本类型范围内唯一，且并发新增不产生重复号。
- **FR-014**（⏸️ 延后，待旧库数据到位）: 系统 MUST 提供数据迁移能力，将老 yadapm 系统中**仅属于批次版本管理**的存量数据导入新系统，迁移**保留原出入库版本号**（不重新生成），其余 yadapm 功能数据不在迁移范围内。
- **FR-015**（⏸️ 延后，待旧库数据到位）: 系统 SHOULD 在迁移数据时，将老系统的年份/批次/产品/任务/提交人员等引用关系映射到新系统对应实体；映射失败的引用 MUST 以可识别方式保留（不静默丢弃）。

### Key Entities *(include if feature involves data)*

- **批次版本（出入库版本）**: 一次批次投产中某子系统的一个软件版本登记。关键属性：出入库版本号（自动生成、唯一）、版本类型、子系统、基准版本号、组包方式、版本状态、版本投产日期、是否涉及 TWS 改造/数据库修改/接口修改、版本说明、备注、提交人员、审计字段。一条版本可关联多个软件中心任务。
- **版本-任务关联**: 批次版本与软件中心任务之间的多对多关系，承载任务号、任务名称、所属项目/需求等回显信息。
- **投产批次**: 版本所属的批次，提供批次号与版本投产日期（来源于新系统既有投产批次实体）。
- **投产年份 / 产品 / 子系统 / 版本类型 / 组包方式 / 版本状态**: 版本登记时的分类/配置维度，部分对应字典、部分对应配置（子系统因含基准版本号等附加属性，倾向独立配置）。
- **旧批次版本数据**: 从老系统迁入的历史批次版本记录，保留原始出入库版本号供查询。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 版本管理人员可在 2 分钟内完成一条批次版本的登记（含级联选择与任务关联），并立即看到正确生成的出入库版本号。
- **SC-002**: 6 种版本类型的出入库版本号生成规则全部正确，针对每种类型的代表性用例 100% 与预期格式一致。
- **SC-003**: 在同子系统+版本类型下连续新增多条版本，出入库版本号序号严格递增、零重复。
- **SC-004**: 批次版本列表按任一查询条件筛选后，结果准确率 100%，常规数据量下结果即时返回（用户感知无明显等待）。
- **SC-005**: 老系统批次版本存量数据迁移后，可在新系统查询到，关键字段与老系统一致率 100%；非批次数据零误迁。
- **SC-006**: 新增/编辑/删除操作均留下可追溯的审计记录，覆盖率 100%。

## Assumptions

- **范围边界**：本 spec 仅覆盖"批次版本管理"单个页面；非批次版本管理、旧数据查询、问题单及缺陷两页属于后续独立 spec，不在本次范围内。
- **数据迁移延后**：本轮**不实现**数据迁移（FR-014/FR-015、User Story 5）—— 旧库最新数据尚未到位，功能先行、迁移后补。功能在空表上即可独立开发与验证。待数据到位后，迁移范围仍严格限定为批次版本管理菜单涉及的存量数据；其余 yadapm 功能（合同、订单、任务、周报等）数据一律不迁；老系统中批次与非批次共用一张物理表，迁移时按"批次"标志筛选。
- **版本号生成规则**：以老 yadapm 系统现行规则为准（6 种版本类型规则见上文场景与需求）；规则的精确细节将在 plan 阶段对照老系统源码逐条核准并以特征测试锁定。
- **出入库版本号不可手改**：沿用老系统行为，版本号完全由系统生成，用户只读。
- **复用新系统既有资产**：投产批次、软件中心任务、产品/年份/排期等概念复用新系统既有实体与字典，不重建任务/批次体系。
- **提交人员默认值**：默认取当前登录用户，允许在表单中选择其他提交人员（沿用老系统可选行为）。
- **权限粒度**：保留老系统"查询 / 编辑 / 删除"分级权限的语义，落地为新系统的权限点。
- **审计方式**：操作审计采用新系统统一的操作日志机制，替代老系统的稽核日志表。
- **附件**：批次版本管理本身不含附件功能（附件属于问题单及缺陷菜单），本 spec 不涉及。
- **产品/子产品归属**（analyze I1 澄清）：「产品」是子系统配置（`pm_sys_name`，含一级产品 p_id + 产品 product）的属性，用于"产品→子系统"级联及类型4 版本号后缀；主表存「子产品」（sub_version_code）用于类型4 的子产品+年份序号维度。两者不同概念，不可混淆。
- **版本状态枚举待生产数据**（analyze A1 澄清）：老系统版本状态枚举值为生产库运行时数据，源码中无 seed。本轮建空字典 `sys_version_status`（管理员可维护），version_status 字段可选；迁移数据到位后灌入真实枚举，届时可收紧为必填。
- **任务关联可空**：录入的软件中心任务号在 `pm_task` 中无对应时，仅保存任务号文本、不强制外键关联。
